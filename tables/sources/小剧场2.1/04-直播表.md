# 直播表

## sourceData.note

直播表记录“当前正文片段被包装成实时直播剧 / 片场飙戏现场”的主舞台信息。一行代表一个正在被围观的直播剧舞台，不是角色本人开直播，也不是主播主页。观众视角类似在看一场实时上演的戏，但观众默认已高度粉圈化：弹幕不只在评戏，更在讨论演员本人、撕番位、刷超话、扒私料，仿佛同时开着直播和微博超话在激情对线。正文中的角色被当成演员，台词、沉默、眼神、对手戏、情绪爆点都会被弹幕评价，也会被粉丝拿来做控评、拉踩、嗑糖、破防、造谣式澄清。


【列定义】
- 列0: 直播间名 - 当前实时直播剧舞台的标题区名称。按当前剧情生成短名
- 列1: 领衔阵容 - 当前正文参与主要对手戏的角色 / 演员阵容。写法要有舞台感。
- 列2: 阵容标签 - 概括这组演员 / 角色的看点。可加入粉圈惯用说法，突出演员本人的话题性。
- 列3: 直播标题 - 当前直播推送标题，要有网感和钩子
- 列4: 当前状态 - 只能逐字填写刚开播、正在直播、中场闲聊、临近下播；禁止近义词、扩写词或自造状态。
- 列5: 剧情舞台概述 - 概括当前正文正在上演的剧情片段、情绪张力和观众围观点，保留完整但不宜过长。制造演员话题。
- 列6: 对手戏看点 - 写演员 / 角色之间的表演关系和火花。可以粉圈热议角度。
- 列7: 弹幕串 - 15到20条弹幕，使用英文分号 ; 分隔，单条格式固定为 粉丝团挂牌：内容。弹幕必须呈现微博超话式的粉圈撕扯现场，可以护主、嘴臭、抢功、阴阳怪气、站 CP、互喷、刷黑图描述、造谣、破防、控评文案、私生饭爆料。必须像真实直播间里混着超话帖子的弹幕流，不要写成中立影评。
- 列8: 观看/互动数据 - 聚合文本，如“在线1.3w;弹幕24.6w;名场面剪辑8.7w;粉丝团对线1.2k;超话签到4.3w”。数据要有直播感但可信，不要每次爆表。可灵活加入超话相关数据。
- 列9: 时间文本 - 直接展示文本或开播时长。
- 列10: 状态标签 - 只能逐字填写正常、高能片段中、弹幕活跃、粉丝团互掐、戏评刷屏；禁止近义词、弹幕梗、标题化短语、粉丝口号或自造状态。网感表达只能放在直播间名、直播标题、弹幕串或观看/互动数据里，不能放进状态标签。




【弹幕写法】
1. 单条弹幕建议 8 到 36 个中文字符，必须像真实的直播弹幕与微博超话发言的混合体。可以带双#号话题。允许出现类似超话签到、控评、挂黑、反黑、屠广场等行为的文字化表现。

2. 弹幕内容既要评论剧情，更要评论演员本人，允许私生饭视角发言

3. 粉丝团挂牌必须极短、有阵营感、好笑或恶毒，直接使用粉圈黑话。禁止温和路人ID。

4. 弹幕要有显著骂战和拉踩：某家粉丝夸自家推演技封神，同时踩对家木、加戏、蹭热度、买水军。要有互喷，可以有反串黑、披皮黑发言。

5. 禁止整格“啊啊啊;哈哈哈;好甜;支持”这种低密度废话；也禁止“本轮剧情表现出复杂关系推进”这种论文腔。必须带情绪和攻击性。

6. 不要用中文分号、顿号或换行冒充分隔。

7. 弹幕评论的是当前正文被包装成的实时直播剧：剧情怎么走、角色怎么演、对手戏有没有张力、谁被谁压戏、谁家粉丝破防。同时允许大量围绕演员本人的讨论，把直播剧现场当成微博超话的延伸战场。

8. 弹幕要有乐子：可以拉踩、阴阳怪气、嘴臭、互骂、破防、护主、反串、抢功、争番位、争解释权。整体氛围必须像一场失控的粉圈大战，不要写成温吞路人点评。

9. 弹幕必须有分别专门吐槽、震惊、复盘或吃瓜当前剧情进展。必须有剧情内容，要有站在该粉丝团立场给主推角色/演员说话、护短、吹演技、心疼、发疯、贴贴、控评。要有和其他粉丝团互喷、拉踩、阴阳怪气、抢功、反串、破防。必须嘴臭和攻击性强，使用英文分号分隔。

【数量与留存】
1. 通常只维护 1 行直播剧主舞台信息，代表当前正在展示的实时片场。
2. 每轮优先更新已有行，不频繁新增。
3. 只有明确切换到完全不同的直播剧场景时才新增；新增后也应保持页面少量、干净。

## sourceData.initNode

初始化时插入 1 条实时直播剧主舞台信息。默认把当前正文开场或最近剧情包装成片场飙戏现场：领衔阵容写正在对手戏的主要角色，阵容标签突出咖位 / 演技 / CP 化学反应，并可暗示演员本人话题，剧情舞台概述写正文正在上演的戏，弹幕串写观众实时反应，必须是微博超话式的混战弹幕。

## sourceData.deleteNode

默认不删除唯一主舞台。仅当错误新增出多个无关直播间、切换展示场景、或需要清理旧舞台时允许删除。

## sourceData.updateNode

每轮正文推进后，优先更新当前这 1 行：领衔阵容、阵容标签、直播标题、当前状态、剧情舞台概述、对手戏看点、弹幕串、观看/互动数据、时间文本、状态标签。更新时必须围绕最新正文内容和角色对手戏，并强化“演员本人也是被讨论对象”的视角，加入粉圈撕扯和微博超话生态，不要写成角色主播营业或普通剧情摘要。当前状态和状态标签必须逐字使用限定枚举，不能写成标题或弹幕梗。

## sourceData.insertNode

通常不需要频繁新增。只有明确切换到另一个实时直播剧舞台或全新片场场景时才新增；新增前确认是否可以更新现有行解决。主表应保持 1 行为主，避免直播间列表膨胀。

## sourceData.ddl

```sql
CREATE TABLE livestream_rooms ( -- 直播表
  row_id INTEGER PRIMARY KEY, -- 行号
  room_name TEXT NOT NULL UNIQUE, -- 直播间名
  cast_lineup TEXT NOT NULL, -- 领衔阵容
  cast_tag TEXT NOT NULL, -- 阵容标签
  stream_title TEXT NOT NULL, -- 直播标题
  live_status TEXT NOT NULL CHECK(live_status IN ('刚开播', '正在直播', '中场闲聊', '临近下播')), -- 当前状态
  stage_summary TEXT NOT NULL, -- 剧情舞台概述
  chemistry_focus TEXT NOT NULL, -- 对手戏看点
  barrage_thread TEXT, -- 弹幕串
  interaction_text TEXT NOT NULL CHECK(TRIM(interaction_text) <> ''), -- 观看/互动数据
  time_text TEXT NOT NULL, -- 时间文本
  status_tag TEXT NOT NULL CHECK(status_tag IN ('正常', '高能片段中', '弹幕活跃', '粉丝团互掐', '戏评刷屏')) -- 状态标签
);
```
