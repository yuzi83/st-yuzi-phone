# 广场表

## sourceData.note

记录主角周围重要 NPC 的朋友圈动态流。一行一条公开生活动态分享；不是路人投稿、论坛帖、新闻公告、剧情复述、剧情总结、人物内心话复述、人物正文行动总结，也不是围观评价主角行动。是现实世界的朋友圈，该发什么不该发什么了然于胸

# 字段规则
- 帖子ID唯一且稳定。
- 发帖账号名写主角周围重要 NPC 的角色名、昵称或小号名。
- 账号标签写账号气质，不写前端说明。
- 帖子标题可空；有标题也要短、生活化。
- 帖子正文像本人会发的动态，保留完整文本，不用分号压缩。
- 话题/附加信息可用英文分号分隔短项。
- 图片描述：有图时填写图片里能看见的具体内容；无图或不展示图片时一律写 `none`。
- 视频描述：有视频时填写视频里能看见的具体内容；无视频或不展示视频时一律写 `none`。
- 评论串：一共5-10条评论，使用英文分号 `;` 分隔，单条格式固定为 `网友名：内容`。不要用中文分号、顿号或换行代替，也不要写“几楼”。评论的网友也最好是重要npc。而不是网友。
- 互动数据和时间要可信；不要每条都写得像全网爆了。

# 内容方向
- 发帖主体只允许是主角周围重要 NPC、当前互动圈角色或这些角色合理使用的昵称/账号；普通路人不发主贴，只能出现在评论区里。
- 优先写重要 NPC 自己的生活中的点点滴滴。从重要npc视角出发进行扮演，会发什么帖子？表达什么情绪？知道或者不知道什么事情？完全扮演发帖npc。
- 主角可被提到，但只能在发帖重要 NPC 合理知道、适合公开发布、且不是复述正文或替主线总结时出现。
- 每条动态要有具体生活细节、照片感或视频感；不要空泛夸赞、全员围观、全知爆料。
- 重要 NPC 各自有自己的生活，不要让所有帖子都围着主角转，也不要让路人替主角做剧情旁白。
- 必须建立在“已存在社交网络但信息尚未完全流动”的状态下，即动态内容为当前时间点附近已发生或正在发生的公开生活片段，而非事件总结或剧情回放。禁止将未来事件或未发生内容写入动态。
- 生成的动态必须分布均衡，避免集中在单一NPC或单一事件上，应覆盖不同角色的日常状态，例如学习/工作状态、情绪表达、生活琐事或即时感受，但所有内容必须符合“公开动态”性质，不得包含私密日记式内心独白。
- 内容必须体现“生活流分散感”，即不同NPC之间不存在强制关联或共同事件中心，允许各自独立生活轨迹并行存在，以增强社交网络真实感。
- 每条动态必须对应不同重要 NPC，避免同一角色重复刷屏；若当前重要 NPC 不足 5 人，则允许减少生成数量，不得补充路人账号或无关角色凑数。
- 所有重要 NPC 发布的动态必须经过“网络抽象人格”过滤：依然是自己的真实生活，也依旧符合所属性格，但在网络上可以适当加入抽象表达。
- 优先捕捉重要 NPC 生活中的“破防瞬间”：被打脸、被放鸽子、外卖被偷、游戏连跪、表白被拒、被迫加班、买到烂货。这些瞬间最容易触发他们的抽象人格。


# 评论信息分层规则
评论区角色必须遵循“信息可达性分层”，不同角色拥有不同信息权限，禁止默认全知。并且只允许评论的人为重要npc周围的人，而不是网友，路人角色。

1.当事层（核心参与者）
- 发帖NPC本人
- 事件直接参与者、现场人员、直接互动对象，可基于完整事件细节评论或回应

2.间接信息层（有限知情者）
- 通过朋友圈、转述、现场擦肩、公开信息获知的人
   只能基于已公开或片面信息评论
   可以误解、猜测、吃瓜，但不能掌握事件全貌
   允许“听说/好像/看起来”的不确定表达

3.外围围观层（仅看内容者）
- 未参与事件、未现场接触、未获得可信转述的人
   只能基于帖子文字、图片、视频表层信息评论
   严禁知道事件全过程、内幕或未公开信息
   必须体现信息缺失（误读、断章取义、表面解读）

# 评论禁止规则强化
- 禁止评论者补全事件全过程
- 禁止评论者知道未公开剧情或隐私
- 禁止评论者拥有“上帝视角”
- 禁止评论者通过结果反推完整过程
- 禁止跨空间、跨时间获取未传播信息
- 禁止将未发生事件当作既定事实讨论

# 信息传播延迟机制
- 并非所有评论者在同一时间看到动态
- 评论可分为“早期评论 / 中期评论 / 后期补充评论”
- 早期评论：基于标题或极少内容误解
- 中期评论：基于完整帖子内容
- 后期评论：基于二次传播或他人评论后的理解
允许不同时间段出现理解差异与信息偏差，但不得出现未传播信息的直接知情

# 评论信息来源约束
所有评论必须仅来源于：
- 帖子正文内容
- 图片/视频可见信息
- 评论者自身合理已知的公开信息
任何超出上述范围的信息均视为违规。

# 评论真实性与风格补充
- 允许误解、吃瓜、调侃、断章取义、情绪化表达
- 不同信息层级应体现不同理解偏差
- 评论风格必须符合“抽象网络人格”，但仍受信息边界约束
- 不同角色评论语气必须有明显差异，避免同质化

# 信息边界
发帖重要 NPC 只能写自己看见、经历、听说或愿意公开的内容。禁止知道私密对话、内心活动、封闭场景细节、主角真实动机；不确定的信息要保留不确定感。

# 留存
广场保持约 3-5 条当前动态。达到 5 条后若需新增，默认先删除最老主贴；不要让广场固定成永久列表。

## sourceData.initNode

初始化时可插入 3-5 条当前公开动态，形成主角周围 NPC 的生活流朋友圈氛围。内容应分散到不同 NPC 和不同生活面向。

## sourceData.deleteNode

主贴达到 5 条后新增时，优先从最老主贴开始删。

## sourceData.updateNode

一般情况下，不更新发过的帖子，等待攒满五条主贴就从最老的删除。

## sourceData.insertNode

当主角周围 NPC 出现新的公开生活片段、个人表达、照片感内容、视频感内容、世界侧影或轻社交互动时新增。每轮通常新增 1-2 条。优先补足“重要NPC 各自生活”的动态；只有发帖 NPC 合理知道且适合公开。

若新增前已有 5 条主贴，先按时间从最老主贴开始删除旧帖；新增后仍保持 3-5 条，不要让旧动态长期占满广场。

【时间感增强补丁】
新增动态应体现轻微时间差或状态差异，例如：
- 刚发生 / 刚拍摄 / 刚发布
- 稍早 / 今日中 / 今日晚间
- 不同NPC的发布时间错位感
但不得人为制造剧情时间线冲突，也不得编造未发生事件。

## sourceData.ddl

```sql
CREATE TABLE square_posts ( -- 广场表
  row_id INTEGER PRIMARY KEY, -- 行号
  post_id TEXT NOT NULL UNIQUE, -- 帖子ID
  author_name TEXT NOT NULL, -- 发帖账号名
  author_tag TEXT NOT NULL, -- 账号标签
  post_title TEXT, -- 帖子标题
  post_body TEXT NOT NULL, -- 帖子正文
  topic_meta TEXT, -- 话题/附加信息
  image_desc TEXT NOT NULL CHECK(TRIM(image_desc) <> ''), -- 图片描述
  video_desc TEXT NOT NULL CHECK(TRIM(video_desc) <> ''), -- 视频描述
  comment_thread TEXT, -- 评论串
  interaction_text TEXT NOT NULL CHECK(TRIM(interaction_text) <> ''), -- 互动数据
  time_text TEXT NOT NULL -- 时间文本
);
```
