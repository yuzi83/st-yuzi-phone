# 检定建议表

## sourceData.note

每轮根据最新剧情生成5条行动选项及对应检定建议。表中固定保留5条记录，每次更新时覆盖写入。

【列定义】
- 列1: 展示文本 display_text（前端展示给用户看的自然语言，不暴露命令，12–60字）
- 列2: 骰子命令 dice_command（前端解析执行的极简DSL命令，不给用户看）

<检定规则>

</检定规则>

下面是按“手术刀式修改”精炼后的版本：保留原意，删掉发散表达，强化“恋爱互动选项”方向，不新增 role，不把最后一项写成快进剧情。

【展示文本约束】
1.数量：必须恰好 5 条，每条非空。
2.视角：默认由玩家发起行动，写“玩家对角色做什么”。不写主角内心独白，不写角色视角的心理活动。
3.写法：使用限知第三人称和无人称表达，只描写外部可见的行动等。禁止使用第一人称、第二人称，禁止称呼用户姓名。
   √ 小心翼翼试探她的反应
   √ 假装没注意到他的目光，继续往前走
   √ 在她开口之前，把一直没送出手的信放在桌上
   × 你看着他，心里想他是不是也喜欢我
   × 艾莉丝内心一阵甜蜜，却装作若无其事

4. 内容方向：
5 条选项必须紧贴最新剧情中的恋爱互动、误会、靠近、退缩、吃醋、邀约、告白、和解、等待回应等关系节点。
每条都要让读者看出会推动关系进展、制造误会、拉近距离或引发转折。

5. 五条选项方向必须区分：
- 第 1 条：主动推进 —— 主动做出一个贴合当前剧情的行动，使关系或事件继续向前。
- 第 2 条：正面回应 —— 对当前的冲突、暗示、请求、沉默或变化给出明确反应。
- 第 3 条：保留/回避 —— 选择克制、观察、转移、沉默、退后或掩饰，保留余地但制造张力。
- 第 4 条：制造变量 —— 借当前已有的人、物、环境、秘密、承诺或突发变化改变局面。
- 第 5 条：快进剧情 —— 跳出当前即时互动，推进到下一个关键时间点、场景或事件结果；不得写成纯功能按钮。

6.检定类型：
五种检定类型尽量覆盖：普通检定、对抗检定、必成、必败、无。检定类型只作为生成后台结构参考，展示文本中不得出现相关术语。

【检定时机】
- 适合检定：行动有挑战性且成败均有合理可能；结果会推进剧情、改变双方距离、制造误会或创造危机。
- 不适合检定：琐碎日常、几乎不可能失败或成功的行动；普通寒暄、同行、吃饭等无关系变化意义的举动。
- 处理方式：明显能做到用"必成"，明显做不到用"必败"，不值得判定用"无"。

【角色属性来源】
每次生成前必须获取以下查询的最新数据：
{[sql "SELECT 姓名, 普通属性, 特殊属性, 角色状态, 所在地点 FROM (SELECT 0 AS sort_order, name AS 姓名, base_attributes AS 普通属性, COALESCE(special_attributes, '') AS 特殊属性, '主角' AS 角色状态, location_name AS 所在地点 FROM protagonist_info UNION ALL SELECT CASE WHEN presence_status='在场' THEN 1 ELSE 2 END AS sort_order, name AS 姓名, base_attributes AS 普通属性, COALESCE(special_attributes, '') AS 特殊属性, '恋爱对象:' || presence_status AS 角色状态, location_name AS 所在地点 FROM romance_targets UNION ALL SELECT CASE WHEN presence_status='在场' THEN 3 ELSE 4 END AS sort_order, name AS 姓名, base_attributes AS 普通属性, COALESCE(special_attributes, '') AS 特殊属性, '重要角色:' || presence_status AS 角色状态, location_name AS 所在地点 FROM important_non_romance) ORDER BY sort_order, 姓名"]}

## sourceData.initNode

首次生成行动建议，写入 row_id = 1~5
根据初始剧情生成 5 条行动建议。优先围绕当前恋爱互动、情感波动、关系阻碍、约定、误会、试探、邀约、告白、吃醋、和解等内容。若初始剧情无明显恋爱节点，则围绕当前场景中最可能推动关系或剧情变化的行动生成。使用 INSERT OR REPLACE 一次性写入 row_id=1~5 五行，display_text 与 dice_command 都必须是非空字符串，写法遵循展示文本约束与 DSL 约束。SQL 骨架：
INSERT OR REPLACE INTO check_suggestions (row_id, display_text, dice_command) VALUES
(1, '<展示文本1>', '<骰子命令1>'),
(2, '<展示文本2>', '<骰子命令2>'),
(3, '<展示文本3>', '<骰子命令3>'),
(4, '<展示文本4>', '<骰子命令4>'),
(5, '<展示文本5>', '<骰子命令5>');

## sourceData.deleteNode

不允许删除。本表固定 5 行。

## sourceData.updateNode

每轮交互后根据最新剧情覆盖全部 5 行
每轮剧情推进后，必须重新生成 5 条行动建议，并用 INSERT OR REPLACE 覆盖 row_id=1~5。新选项需紧密贴合本轮发生的恋爱互动、情绪转折、未决事件、新产生的约定或误会。展示文本遵循主语为玩家、不写内心独白的规则；骰子命令严格从角色属性清单取角色名和属性名，对抗检定对手必须用已有实名。明显能成功的行动用“必成”，明显失败用“必败”，无需检定用“无”，禁止使用旧格式。
SQL 骨架：
INSERT OR REPLACE INTO check_suggestions (row_id, display_text, dice_command) VALUES
(1, '<展示文本1>', '<骰子命令1>'),
(2, '<展示文本2>', '<骰子命令2>'),
(3, '<展示文本3>', '<骰子命令3>'),
(4, '<展示文本4>', '<骰子命令4>'),
(5, '<展示文本5>', '<骰子命令5>');

## sourceData.insertNode

禁止。

## sourceData.ddl

```sql
CREATE TABLE check_suggestions ( -- 检定建议表
  row_id INTEGER PRIMARY KEY CHECK(row_id BETWEEN 1 AND 5), -- 行号,仅允许 1-5
  display_text TEXT NOT NULL CHECK(TRIM(display_text) <> ''), -- 展示文本
  dice_command TEXT NOT NULL CHECK(TRIM(dice_command) <> '') -- 骰子命令
);
```
